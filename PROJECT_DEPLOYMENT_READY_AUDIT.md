# Полный Аудит Проекта и Подготовка к Деплою (Portfolio)

В данном документе собран глубокий анализ текущего состояния вашего full-stack проекта (Django + Next.js), включающий разбор проблем с отправкой email, оценку архитектуры, а также строгий аудит безопасности перед выходом в Production.

---

## 🛠 1. Почему не отправляется электронная почта?

Вы упомянули, что у вас не получается отправить email, и предположили, что это может быть связано с использованием локального сервера. **Вы абсолютно правы, это одна из самых частых проблем.** 

Давайте разберем основные причины, почему это происходит у вас, и как это исправить:

### Причина А: Блокировка SMTP-портов провайдером (ISP)
Большинство домашних интернет-провайдеров (если вы запускаете сервер с домашнего Wi-Fi) по умолчанию блокируют исходящие соединения на порты `25` и часто `465` (который вы используете для Яндекса), чтобы предотвратить рассылку спама с зараженных компьютеров.
**Решение:** При деплое на реальный VPS/VDS (например, Ubuntu сервер в облаке) эти порты будут открыты, и скрипт отработает успешно. 

### Причина Б: Политика безопасности Яндекса (Пароль приложения)
В вашем `backend/.env` указан `EMAIL_HOST_PASSWORD=sghmxaebbnvdgvuo`. Если это ваш **основной** пароль от учетной записи, Яндекс отклонит авторизацию.
**Решение:** Вам необходимо зайти в аккаунт Яндекса → Безопасность → Пароли приложений и создать специальный пароль для "Почты", и именно его использовать в `.env`. Если вы уже используете пароль приложения, то проблема скорее всего в провайдере (Причина А).

### Причина В: Молчаливое игнорирование ошибок (Скрытая проблема кода)
В вашем файле `api/views.py` (строки 62-72) отправка почты завернута в блок `try/except`:
```python
try:
    send_mail(..., fail_silently=False)
except Exception as e:
    # Ошибка ловится, показывается в консоли, а форма на фронтенде показывает Success
    print(f"SMTP Error: {e}")
```
**Решение:** Загляните в консоль или терминал, где запущен `python manage.py runserver`, в момент отправки формы. Там будет напечатана **конкретная** причина отказа (например, `Authentication failed` или `Connection refused`). Для дебага это лучший способ узнать правду.

## 🔒 2. Аудит безопасности (Подготовка к Deploy)

Перед тем как выкатывать проект на реальный сервер (VPS/VDS), **обязательно** устраните следующие критические дыры в безопасности:

### 🔴 КРИТИЧЕСКИЕ (Исправить до деплоя)
1. **Утечка SECRET_KEY**
   - **Где:** `backend/core/settings.py`
   - **Суть:** В файле настроек `SECRET_KEY` прописан в открытом виде (строка 29).
   - **Решение:** Вынести его в `backend/.env`: `SECRET_KEY = os.environ.get('SECRET_KEY')` и сгенерировать новую строку для продакшена.
2. **Пароль БД в `.env` слишком короткий**
   - **Где:** `backend/.env`
   - **Суть:** Пароль `123` для базы `USER_PASSWORD_BD` — это гарантированный взлом при первой же попытке сканирования порта 5432, если он торчит наружу.
   - **Решение:** Сгенерировать криптографически стойкий пароль (от 16 символов).
3. **Режим DEBUG включен**
   - **Где:** `backend/core/settings.py` (строка 32)
   - **Суть:** `DEBUG = True` оставит ваш проект уязвимым к раскрытию исходников при вызове ошибки.
   - **Решение:** Сделать `DEBUG = os.environ.get('DEBUG', 'False') == 'True'`. Для продакшена в `.env` нужно добавить `DEBUG=False`.

### 🟠 ВЫСОКИЙ ПРИОРИТЕТ
1. **Отсутствие Production URL в ALLOWED_HOSTS и CORS**
   - **Где:** `backend/core/settings.py`
   - **Проблема:** Сейчас там жестко прописаны `localhost`. При деплое на публичный домен, Django заблокирует все запросы с ошибкой 400 (Bad Request).
   - **Решение:** Использовать переменные окружения: `ALLOWED_HOSTS = os.getenv('ALLOWED_HOSTS', 'localhost').split(',')`.
2. **Hardcode API URL на Frontend**
   - **Где:** `frontend/src/lib/api.ts` и `frontend/src/components/ContactTerminal.tsx`
   - **Проблема:** URL для связи с API жестко прописан как `http://127.0.0.1:8000/api`. При сборке Next.js (`npm run build`) этот URL сохранится, и сайт в интернете будет пытаться обратиться к локальному компьютеру посетителя.
   - **Решение:** Использовать переменную. В Next.js это переменная с префиксом: `const BASE_URL = process.env.NEXT_PUBLIC_API_URL || "http://127.0.0.1:8000/api";`

## 🚀 3. Общие архитектурные улучшения

1. **Защита от спама (Rate Limit)**
   - В вашем `ContactTerminal.tsx` и `views.py` уже реализован `ContactRateThrottle` (5 сообщений в день). Это круто! Убедитесь, что при деплое у вас корректно работает распознавание IP, иначе за балансировщиком нагрузки (например, Nginx) все пользователи будут считаться одним IP. Для Nginx вам в Django нужно будет проверять заголовок `X-Forwarded-For`.
2. **Кэширование в Next.js (Revalidation)**
   - Вы используете `next: { revalidate: 3600 }` (кэш на 1 час). Для портфолио это хорошая и бережливая практика, но учтите, что после добавления нового проекта в Django админке, вам придется ждать до часа, чтобы он появился на сайте. Вы можете внедрить **On-Demand Revalidation** (по Webhook из Django), чтобы данные обновлялись мгновенно ровно в момент сохранения в админке.
3. **Обманчивое поле `encrypted_payload`**
   - В базе данных и контактной форме данные называются "encrypted_payload", но по факту они ничем не шифруются (разве что вы планируете шифровать их позже или это просто стилизация "под хакера"). Если это не было задумано как стилизация, то лучше переименовать в просто "message" или добавить честную криптографию на стороне отправляющего юзера (PGP).

### Итог по шагам для Deploy:
1. Вынести все пароли и ключи из `.py` в `.env`.
2. Собрать Next.js (`npm run build`).
3. Настроить Nginx в качестве Reverse-Proxy.
4. Выпустить SSL-сертификаты (Certbot / Let's Encrypt), так как браузеры блокируют `fetch() ` запросы со страниц HTTPS на HTTP API.
5. Настроить Gunicorn или uWSGI для запуска Django (не используйте `runserver` в проде!).
